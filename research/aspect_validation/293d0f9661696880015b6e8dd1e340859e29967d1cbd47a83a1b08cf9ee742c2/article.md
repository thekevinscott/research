## The discipline

The post is widely credited as naming the practice:

> "Three months after OpenAI published a write-up on how their engineers actually shipped production agents (using guides, sensors, and structured constraint files), a name emerged for the discipline: harness engineering."

Key constructs (paraphrased from snippets):

- **Guides** — long-form natural-language documents that orient the agent on architectural intent.
- **Sensors** — automated checks that report environment / runtime / behavior facts the agent can read.
- **Structured constraint files** — machine-checkable rules the harness enforces between agent turns.

## April 2026 update

> "OpenAI's April 2026 update added native sandbox execution, configurable memory, and sandbox-aware orchestration to the Agents SDK, with a shift toward a 'model-native harness' that aligns execution patterns with how frontier models actually perform best."

This is significant: it suggests the platform evolves *toward* the harness pattern (sandbox, memory, orchestration) rather than the harness being a bolt-on. "Model-native harness" is OpenAI's framing.

## Why it matters

This is the OpenAI-side analog of Datadog's harness-first post and StrongDM's Software Factory. Three-person team, multi-month duration, zero human-written code is the first public case study from a frontier lab matching StrongDM's claim. Together with the Anthropic eval post and Datadog's harness post, it forms a roughly consistent industry doctrine.
