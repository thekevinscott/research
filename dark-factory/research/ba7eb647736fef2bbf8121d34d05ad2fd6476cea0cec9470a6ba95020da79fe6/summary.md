# StrongDM Software Factory — The Principles

The principles page distills StrongDM's Dark Factory doctrine into a single closed loop: **Seed → Validation harness → Feedback loop, with tokens as the fuel**. A "seed" is whatever boots the system — a PRD, a few sentences, a screenshot, an existing codebase — and the loop runs until holdout scenarios pass and stay passing. The harness must be end-to-end and as close to the real customer/integration/economic environment as possible; the feedback step samples output and feeds it back into inputs so the system self-corrects and compounds correctness.

The page is short but operational: it enumerates the substrates StrongDM treats as scenario sources — traces, screen captures, conversation transcripts, incident replays, adversarial use, agentic simulation, just-in-time surveys, customer interviews, price-elasticity testing — making explicit that "validation" in a software factory is broader than a test suite. The recurring mantra "Apply More Tokens" is recast as a forcing function: for every obstacle, ask how to convert it into a representation the model can understand and then spend tokens on it. This is the canonical primary-source statement of how StrongDM thinks about the loop.
