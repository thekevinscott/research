# Pulumi Blog — The Dark Factory Pattern for Infrastructure: Running Pulumi Lights-Out

A vendor case study extending the Dark Factory pattern from application code to infrastructure-as-code. Pulumi's post opens by surveying the canonical 2026 examples — StrongDM shipping ~32k lines without writing or reviewing any of it, Stripe's Minions merging 1,000+ PRs/week, Shapiro's five-level ladder, BCG's "dark software factory" framing — and then argues that infrastructure adds blast radius, drift, secrets, irreversible actions, and multi-region state on top of the application case. A code factory shipping a broken UI is a bad UX; an infra factory shipping a broken IAM policy is a postmortem.

The piece's contribution is a concrete four-layer infra-factory architecture: spec, generator, validator, executor. It pulls StrongDM's holdout-scenarios pattern over to IaC (plain-English BDD acceptance tests stored where the generator cannot reach them, with each scenario run three times against an ephemeral deployment, and a 90% pass-rate gate). Pulumi's components map onto these layers — Automation API as the engine, ESC/OIDC for short-lived credentials, CrossGuard for deterministic policy at preview time, Pulumi Cloud Deployments as a governed runner, and Pulumi Neo's three modes (Auto/Balanced/Review) explicitly aligned to Shapiro's levels 5, 4, 3. It's both a vendor pitch and a serviceable infrastructure-flavored variant of the StrongDM playbook.
